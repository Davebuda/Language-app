## What this plan addresses

The repair loop **does inject items correctly** — the architecture works. Wrong answer → explanation card → "Prøv igjen" → injects [micro-drill1, micro-drill2, retry] after the current item → advances to micro-drill1. The walkthrough saw this as "repair loop not working" because the session counter grew (14→17) and Q2 was a different exercise, but that IS the correct flow.

Two defects in the injected retry item specifically:

**Defect 1 — retry uses a random sentence, not the original.**  
`ErrorLogEntry.sentenceId` already exists as an optional field in the type but is never populated. In `makeRepairItems`, the retry item's `contentId` falls through to `getSentence()` (random pool draw). However, there is a second problem: even if `sentenceId` were populated in the error object, `resolveItem` in `useSession.ts` ignores `contentId` entirely — it always picks from the concept seed pool by concept ID. Threading `sentenceId` to `contentId` alone is not sufficient.

**Defect 2 — retry uses a different exercise type, not the original.**  
`buildRepairPlan` intentionally computes a "slightly different" type (`fill-in-blank` original → `sentence-transformation` retry, etc). This serves the micro-drills, not the retry. The micro-drills already vary the exercise type. The retry's job is verification, which requires the original type.

## The change

### Step 1 — `src/types/session.ts`: add `sentenceId` to `ExerciseResult`

```typescript
export interface ExerciseResult {
  sessionId: string;
  itemId: string;
  correct: boolean;
  userAnswer: string;
  correctAnswer: string;
  timeTakenSeconds: number;
  errorTag?: ErrorTag;
  conceptId: string;
  sentenceId?: string;  // ← add: sentence that triggered this result; used by retry step
}
```

This also lets the fingerprint's `recentErrors` (which store `ErrorLogEntry`) record which sentence caused each error — useful for future diagnostics. `ErrorLogEntry` already has `sentenceId?: string`; we just never populated it.

### Step 2 — `src/hooks/useSession.ts`: capture sentenceId in `submitResult`

The resolved content for the current item is already in `contentCache.current` by the time the user submits. Read it there — no change needed in exercise components.

In `submitResult`, after reading `item`:

```typescript
const currentIndex = useSessionStore.getState().currentItemIndex;
const item = useSessionStore.getState().session?.items[currentIndex];
const resolvedSentenceId = contentCache.current.get(item?.id ?? '')?.id;
```

Then include it in the error object:

```typescript
const error = {
  id: crypto.randomUUID(),
  conceptId: result.conceptId,
  errorTag: result.errorTag ?? 'unspecified',
  exerciseType: item?.exerciseType ?? 'translation-to-norwegian',
  wrong: result.userAnswer,
  correct: result.correctAnswer,
  timestamp: new Date().toISOString(),
  sentenceId: resolvedSentenceId,  // ← add
} as const;
```

Why capture from `contentCache` rather than from `ExerciseResult`? The exercise components would each need updating (5 components). The contentCache approach is a single change in one place and handles both seed and AI-generated sentences.

### Step 3 — `src/hooks/useSession.ts`: pre-seed retry item's cache in `continueAfterRepair`

**The critical finding:** `resolveItem` ignores `contentId`. It picks by concept ID from the seed pool. Threading `sentenceId` to the error object and then to `makeRepairItems`'s retry `contentId` is necessary for fingerprint logging but NOT sufficient to make the retry show the original sentence. `resolveItem` will still pick randomly.

The fix: pre-seed the retry item's `contentCache` entry with the original resolved content immediately after injecting the repair items. `resolveItem` has an early return when the item is already cached (`if (contentCache.current.has(item.id)) return`), so pre-seeding prevents the random pool draw.

```typescript
// In continueAfterRepair, after makeRepairItems and before exitRepair:

const originalContent = contentCache.current.get(currentItem.id);
const retryItem = repairItems[repairItems.length - 1]; // last item is always the retry
if (originalContent && retryItem) {
  contentCache.current.set(retryItem.id, originalContent);
}
```

This works regardless of whether the original content was a JSON seed or AI-generated — `contentCache` holds the resolved content directly.

### Step 4 — `src/engine/repair-loop.ts`: `buildRepairPlan`

Replace the "slightly different exercise type" logic:

```typescript
// Current — remove this:
const retryType: ExerciseType =
  error.exerciseType === 'fill-in-blank'
    ? 'sentence-transformation'
    : error.exerciseType === 'sentence-transformation'
      ? 'translation-to-norwegian'
      : 'fill-in-blank';

// Replace with:
const retryType: ExerciseType = error.exerciseType;
```

Remove the comment `// Retry is a slightly different exercise type from the original` — it described the old intent that is now reversed by design decision.

## Edge cases

| Scenario | Behavior |
|---|---|
| Learner wrong on the retry | `!currentItem.isRepairItem` guard in `continueAfterRepair` prevents further repair loop injection — existing behavior, no change |
| Original content was AI-generated (not in JSON corpus) | Pre-seeding from `contentCache.current` works for any resolved content regardless of source |
| `contentCache.current.get(currentItem.id)` returns `undefined` | Only possible if the exercise was never shown (shouldn't happen — resolveItem runs before the item renders). If it does happen: `originalContent` is undefined, the pre-seed branch is skipped, retry falls back to `resolveItem`'s normal pool draw — graceful, no crash |
| Session ends before learner reaches retry item | Normal session completion — the injected items are never reached, which is fine |
| Retry exercise type is `listening-comprehension` or `dictation` | These types require audio and will silently fall back to the empty ListeningExercise rendering. The real fix for listening exercises is out of scope here. Worth a `console.warn` if `retryType` is a listening type with no audio. Not a blocker for item 3 — listening seeds are excluded by the scheduler guard anyway. |

## Acceptance criteria

1. Submit a wrong answer on a `translation-to-norwegian` exercise for concept A using sentence S.
2. "Prøv igjen" is clicked → micro-drill1 appears (different sentence, different type).
3. Micro-drill1 completes → micro-drill2 appears.
4. Micro-drill2 completes → retry appears: **same sentence S, same exercise type `translation-to-norwegian`**.
5. Correct answer on retry → session advances to the next scheduled item.
6. Wrong answer on retry → session advances (no recursive repair loop).
7. TypeScript strict mode passes. Unit tests pass.
