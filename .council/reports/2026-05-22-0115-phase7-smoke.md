### ⚠️ P1 — React error #418 on /dashboard (both breakpoints, both navigations)
**Symptom:** Single console error message:
```
Uncaught Error: Minified React error #418
visit https://react.dev/errors/418?args[]=text&args[]= for full message
```
React error 418 = "Text content does not match server-rendered HTML" (hydration mismatch on a text node).

**What this means:** Some text rendered on the server doesn't match what the client renders during hydration. The minified production message has empty args so the offending text cannot be identified from prod alone.

**Likely sources (priority order to investigate):**
1. `todayFormatted()` in `src/app/dashboard/page.tsx:64-68` — uses `new Date().toLocaleDateString('nb-NO', ...)`. Server timezone (likely UTC) vs client timezone (likely CET / nb-NO) can render different day names. The "FREDAG 22. MAI" header at top is the likely culprit.
2. `getStreak()` from `@/lib/streak` — reads localStorage on client; server renders the fallback 0; client may render the cached value.
3. `WeekStrip`'s `daysActiveLabel` — uses `Date.now()` and `weekStartedAt` for elapsed-days math. Should not SSR because component is `'use client'`, but Next.js still pre-renders client components for the initial HTML payload. If `weekStartedAt` differs server-vs-client (which it does — `null` on server, populated after bootstrap on client), the WeekStrip would render different content. But the parent guard `{fingerprint ? <WeekStrip /> : null}` should prevent SSR rendering of WeekStrip — on the server `fingerprint` is null.

**Not caused by Phase 6.** Even if WeekStrip's empty-state copy were the source, the guard above means it never reaches the server-rendered HTML. The error is almost certainly pre-existing in the dashboard's date-formatted header or streak counter.

**Recommendation (out of Phase 7 scope, for a follow-up):**
- Reproduce locally in `npm run dev` (unminified React) to get the actual text mismatch.
- Most likely fix: wrap `todayFormatted()` and `getStreak()` calls in a `useState/useEffect` to defer the value until after hydration, OR use `suppressHydrationWarning` on the specific date span (sparingly).
- Alternatively, set the Next.js server timezone explicitly via `process.env.TZ=Europe/Oslo` if the server isn't already on CET.
